Introduction

Metropolis & Yellow Office are two architecture firms that solicited my collaboration in developing a system that allowed them to create and manage projects that are part of their portfolio. By collaborating with them I took on a challenge, especially when it came to back-end development.
Making abstraction of the digital, interaction and experience design, the biggest milestones where marked by learning new development techniques in a short amount of time. Some of these were: security (hashing, encrypting, spam prevention, session tokens), creating a dynamic & editable grid system, developing asynchronous multiple-file upload mechanisms or designing a light-weight responsive system for mobile devices.

The process

From the beginning, the clear direction of the project was established by the strategic goal of the firms, which was international expansion. The interaction concept had to be created from a few basic assumptions of functionality, without many specific requirements stated.
The first step was planning and communicating the chosen direction. Being on the same page with the concept was essential, mainly because it required the clients to participate in concept development and give feedback along the way. Afterwards this step, the first back-end sketches were created. Following a sketch ~ wireframe ~ mock-up ~ prototype ~ shipped functionality 'system', all of the backend was delivered in a agile manner. Prior to the first back-end beta, the first database architecture (ER Diagram) was drafted.
The next step included the development of the front-end, which was done in a similar manner. It was then shipped as a demo.
Finally, after bug fixes a final product was ready for launch. See some of the development notes.

The back-end

The back-end is a step-by-step management hub, a custom light-weight platform that allows fast adding or editing of content. The primary feature is quick-storage of a project in the system.

Adding a project is done in 4 steps:

  • Inserting the project information

    The first stage requires the addition of basic information such as name, type and location.

    The information is automatically reused for fields that do not usually require the user's attention.
    In some cases, they are even beyond the user's competences, therefore they are being filled before he reaches them. For example, the project url, Facebook Meta and Date are automatically generated based on information previously entered:


  • Adding the project images, sorting and choosing the background

    The addition of images is limited to a minimum amount of steps. Only selecting a series of images starts the upload and in a matter of seconds they are displayed as thumbnails underneath.

    Furthermore, images can be uploaded on multiple turns. The user is also allowed to manipulate them right after upload:

    Example of deletion:

  • Positioning the project thumbnail on the 'grid'

    The grid is in fact the homepage of the website. It is a responsive grid that reacts to screen rotation or resizing and can be maximized to full screen (details will be shown at the front-end chapter).

    The user is given a random grid position that can be manually changed. Further editing the grid can be done also after the project has been uploaded.

  • Publishing the project, displaying as news and other checkboxes

    Finally, the last step consists of the actual submission of project data. A few options can be 'ticked' and are later editable from the project manager.


The steps above are built in a responsive manner, therefore allowing projects to be added via mobile device.


The parent (from the database point of view - the projects have 'category_id' as a foreign key, 1 to many relation) container of the projects 'categories' is also manageable in the backend. The categories screen allows fast access of add or delete commands and offers an overview of the existing projects. Projects can be filtered by category and status.

A thing to note: throughout the back-end basic user errors are considered in order to create a better user experience. For example, in this case categories that contain projects cannot be deleted, preventing an action that will result in the deletion of multiple projects. All project info can be edited from the project manager page. Leaving the CRUD features aside, a "publish/unpublish" button is present for fast switching of the projects state.

User errors are prevented for certain actions by requiring confirmations (for example, deletion).

Before leaving the back-end description, it is worth saying that it's architecture was built to be scalable and extendable. The platform had to be customized for both organizations, Metropolis and Yellow Office. Each of the organizations had different projects with small specific front-end requirements - a tailored system for their needs.

The front-end

The back-end grid that you have already seen is present on the front-page and the view can be toggled to full screen by collapsing the menu on the left.

An overview of the created projects is available on the projects page, where they can also be filtered.

The front-end is built on a fluid grid system, using the entire screen canvas to display beautiful imagery.
Pages take advantage of high resolution images, having a content-first approach. As with the back-end, views are adapted for lower resolutions.

tl, dr

The challenge behind this project was to create an extendeble system that 'supports' the new expansion strategies followed by the two client organizations. A close collaboration with the firms resulted in the development of a small in-house CMS, with a responsive back-end and front-end design that is easy to update.
